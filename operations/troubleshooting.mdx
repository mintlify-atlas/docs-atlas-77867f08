---
title: 'Troubleshoot luci-app-mosdns issues'
description: 'Fix common luci-app-mosdns problems including service not starting, DNS not resolving, DNS hijacking failures, geodata errors, and cache issues.'
---

Most luci-app-mosdns issues fall into a small number of categories: the service not starting, DNS queries not reaching MosDNS, or data files being missing or stale. The sections below walk through each problem with specific commands to check and steps to resolve it.

## Useful diagnostic commands

Run these commands over SSH to gather information before diving into a specific issue.

<CodeGroup>

```shell Check UCI config
uci show mosdns
```

```shell Check service status
/etc/init.d/mosdns status
```

```shell View logs
cat /var/log/mosdns.log
```

```shell Check nftables DNS hijack table
nft list table inet mosdns
```

```shell Check dnsmasq server config
uci get dhcp.@dnsmasq[0].server
```

```shell Check v2dat extracted files
ls /var/mosdns/
```

```shell Check running processes
ps | grep mosdns
```

</CodeGroup>

## Common issues

<AccordionGroup>

  <Accordion title="Service won't start">
    MosDNS silently exits when `enabled` is set to `0` in UCI. This is the most common reason the service appears to do nothing after installation.

    **Check the enabled flag:**

    ```shell
    uci get mosdns.config.enabled
    ```

    If the output is `0`, enable the service:

    ```shell
    uci set mosdns.config.enabled='1'
    uci commit mosdns
    /etc/init.d/mosdns start
    ```

    **Check the log for errors:**

    ```shell
    cat /var/log/mosdns.log
    logread | grep mosdns
    ```

    Startup errors — such as a missing or malformed config file — appear in the procd log before the log file is written.

    **Verify the config file exists:**

    ```shell
    ls -la /var/etc/mosdns.json
    ```

    If the file is missing, restart MosDNS to regenerate it:

    ```shell
    /etc/init.d/mosdns restart
    ```
  </Accordion>

  <Accordion title="DNS queries not going through MosDNS">
    MosDNS listens on port 5335 by default. For queries to reach it, dnsmasq must be configured to forward to `127.0.0.1#5335`.

    **Check DNS forward is enabled in LuCI:**

    Go to **Services → MosDNS → Basic Setting** and confirm **DNS Forward** is toggled on.

    **Check dnsmasq server setting:**

    ```shell
    uci get dhcp.@dnsmasq[0].server
    ```

    The output should include `127.0.0.1#5335` (or whichever port you configured).

    **Test resolution directly:**

    ```shell
    nslookup example.com 127.0.0.1
    dig @127.0.0.1 -p 5335 example.com
    ```

    If the direct query to port 5335 succeeds but normal DNS fails, the dnsmasq forwarding rule is missing. Save and apply the MosDNS settings again to re-apply the dnsmasq configuration.
  </Accordion>

  <Accordion title="DNS hijacking not working">
    DNS hijacking intercepts all outbound port 53 UDP traffic using an nftables NAT rule. This feature requires **fw4** (OpenWrt 22.03 and later) and will not work on older builds that use iptables.

    **Check that DNS redirect is enabled in LuCI:**

    Go to **Services → MosDNS → Basic Setting** and confirm **DNS redirect** is toggled on.

    **Check that the nftables table was created:**

    ```shell
    nft list table inet mosdns
    ```

    If the command returns an error such as `Error: No such file or directory`, the table was not created. This usually means fw4 is not present or the service started before the firewall was fully initialized.

    **Confirm fw4 is available:**

    ```shell
    ls /sbin/fw4
    ```

    If `/sbin/fw4` does not exist, your OpenWrt version does not support DNS hijacking. Use **DNS Forward** instead, which routes dnsmasq queries to MosDNS without requiring nftables.

    Restart MosDNS after confirming fw4 is present:

    ```shell
    /etc/init.d/mosdns restart
    nft list table inet mosdns
    ```
  </Accordion>

  <Accordion title="GeoSite/GeoIP dat files missing">
    MosDNS reads extracted `.txt` files from `/var/mosdns/` at startup. These are generated by the `v2dat` tool from the source `.dat` files in `/usr/share/v2ray/`. If extraction fails, MosDNS cannot load its routing rules.

    **Check for the extracted text files:**

    ```shell
    ls /var/mosdns/
    ```

    You should see files such as `geoip_cn.txt`, `geosite_cn.txt`, and `geosite_geolocation-!cn.txt`. If the directory is empty or missing these files, run v2dat manually:

    ```shell
    /usr/share/mosdns/mosdns.uc v2dat_dump
    ```

    **Check the source dat files exist:**

    ```shell
    ls /usr/share/v2ray/
    ```

    If `geoip.dat` or `geosite.dat` are missing, go to **Services → MosDNS → Geodata Update** and click **Check And Update** to download them.
  </Accordion>

  <Accordion title="Geodata update fails">
    The geodata update downloads `geoip.dat` and `geosite.dat` from GitHub. Network connectivity problems or GitHub access restrictions cause update failures.

    **Check network connectivity:**

    ```shell
    curl -I https://github.com
    ```

    **Try enabling the GitHub proxy:**

    Go to **Services → MosDNS → Geodata Update** and enter a GitHub proxy URL in the **GitHub Proxy** field (for example, `https://gh-proxy.com`). Click **Check And Update** again.

    **Check the log for details:**

    ```shell
    cat /var/log/mosdns.log
    logread | grep mosdns
    ```

    Download errors and checksum failures are written to the log during the update process.
  </Accordion>

  <Accordion title="Ad block rules not downloading">
    The ad block rule downloader writes a lock file at `/var/lock/mosdns_ad_update.lock` while it runs. If a previous download was interrupted, the lock file may remain and block future updates.

    **Check for a stale lock file:**

    ```shell
    ls /var/lock/mosdns_ad_update.lock
    ```

    If the file exists and no update is currently running, remove it:

    ```shell
    rm /var/lock/mosdns_ad_update.lock
    ```

    **Check that the ad source URLs are reachable:**

    ```shell
    curl -I <your-ad-source-url>
    ```

    If you are using a raw GitHub URL, enable the **GitHub Proxy** option in **Geodata Update** settings to route the download through a mirror. The proxy setting applies to both geodata and ad block rule downloads.

    After removing the lock file, restart MosDNS to trigger a fresh download:

    ```shell
    /etc/init.d/mosdns restart
    ```
  </Accordion>

  <Accordion title="TLS certificate errors">
    MosDNS validates the TLS certificates of DoH and DoT upstream servers by default. If the system CA certificates are expired or the system clock is significantly wrong, certificate validation fails and upstream queries are rejected.

    **Symptoms:** Queries time out or return SERVFAIL. The log shows TLS handshake errors.

    **Temporary fix — disable certificate validation:**

    Go to **Services → MosDNS → Basic Setting → Advanced Options** and enable **Disable TLS Certificate** (`insecure_skip_verify`). Click **Save & Apply**.

    <Warning>
    Disabling TLS certificate validation removes a security control. Only use this as a short-term workaround while you fix the underlying certificate or clock issue.
    </Warning>

    **Fix the system clock:**

    ```shell
    /etc/init.d/sysntpd restart
    date
    ```

    Once the clock is correct, disable the **Disable TLS Certificate** option and restart MosDNS.
  </Accordion>

  <Accordion title="Cache not working">
    The DNS cache is enabled by default but will not function if it is disabled in the config or if the affected domains are excluded from caching by design.

    **Check cache is enabled:**

    ```shell
    uci get mosdns.config.cache
    uci get mosdns.config.cache_size
    ```

    Both `cache` should be `1` and `cache_size` should be greater than `0`.

    **Domains excluded from caching by design:**

    MosDNS deliberately skips the cache for:
    - Domains in **DDNS Lists** — these use a forced 5-second TTL and are not cached.
    - Domains in **Block Lists** or **Ad block** rules — these are rejected without caching.
    - PTR queries for local pointer domains.

    If a domain you expect to be cached falls into one of these categories, remove it from the relevant rule list.

    **Flush and re-test:**

    ```shell
    curl http://127.0.0.1:9091/plugins/lazy_cache/flush
    ```

    Then query the same domain twice and check the log for a cache hit on the second query.
  </Accordion>

  <Accordion title="Custom config mode: Rules tab has no effect">
    When you select **Custom Config** in Basic Setting, MosDNS uses the YAML file at `/etc/mosdns/config_custom.yaml` instead of the auto-generated configuration. The **Rules** tab in LuCI writes rule files that are only referenced by the default configuration.

    If you are in custom config mode, edit your YAML file directly to add or modify rules. The whitelist, blocklist, greylist, DDNS list, hosts, redirect, and other rule files are not automatically included in a custom config unless you reference them explicitly.

    To switch back to the default configuration, go to **Basic Setting**, set **Config File** to **Default Config**, and click **Save & Apply**. Rule settings will take effect again after the restart.
  </Accordion>

  <Accordion title="Dnsmasq settings not restored after disabling MosDNS">
    When MosDNS enables DNS forwarding, it saves the original dnsmasq settings to a lock file at `/etc/mosdns/redirect.lock` and restores them when you disable the service. If this file is missing or corrupted, the restore step is skipped and dnsmasq may be left pointing at MosDNS on port 5335.

    **Check for the lock file:**

    ```shell
    ls -la /etc/mosdns/redirect.lock
    cat /etc/mosdns/redirect.lock
    ```

    **If the file is stuck or corrupted, delete it manually:**

    ```shell
    rm /etc/mosdns/redirect.lock
    ```

    Then disable MosDNS and restart dnsmasq to restore the original DNS settings:

    ```shell
    uci set mosdns.config.enabled='0'
    uci commit mosdns
    /etc/init.d/mosdns stop
    /etc/init.d/dnsmasq restart
    ```

    Verify dnsmasq is using its original upstream servers:

    ```shell
    uci get dhcp.@dnsmasq[0].server
    ```
  </Accordion>

</AccordionGroup>
